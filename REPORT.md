# Отчет по лабораторной работе №1

## Тема
Преобразование исходных данных в аналитическую модель типа «снежинка».

## Цель работы
Цель работы - загрузить исходные CSV-файлы в PostgreSQL, разобрать сырую таблицу на отдельные сущности и построить модель данных, которую удобно использовать для аналитических запросов.

Исходные данные описывают продажи товаров для домашних питомцев. В одной строке файла находятся данные о продаже, покупателе, продавце, магазине, товаре и поставщике. Поэтому перед построением аналитической модели эти данные нужно разделить на таблицу фактов и таблицы измерений.

## Загрузка исходных данных
В скрипте `01_init_source.sql` создается таблица `source_data`. В нее загружаются все 10 CSV-файлов. Каждый файл содержит 1000 строк, всего получается 10000 строк.

После загрузки в таблицу добавляется поле `source_row_id`. Это технический идентификатор строки, который нужен для связи исходных данных с уже преобразованными данными.

## Построенная модель
В результате была построена модель типа «снежинка». Основная таблица фактов хранит продажи, а описательные данные вынесены в измерения и справочники.

### Таблица фактов
`fact_sales` хранит данные о продажах:
* дата продажи;
* покупатель;
* продавец;
* товар;
* магазин;
* количество проданного товара;
* сумма продажи.

### Основные измерения
* `dim_customer` - покупатели;
* `dim_seller` - продавцы;
* `dim_store` - магазины;
* `dim_product` - товары;
* `dim_supplier` - поставщики;
* `dim_date` - календарь продаж.

### Справочники
Для нормализации отдельно вынесены:
* `dim_country` - страны;
* `dim_pet_type` - виды питомцев;
* `dim_pet_breed` - породы питомцев;
* `dim_pet_category` - категории питомцев для товаров;
* `dim_category` - категории товаров.

## Особенности преобразования
При просмотре данных было видно, что исходные идентификаторы вроде `sale_customer_id`, `sale_seller_id` и `sale_product_id` повторяются в разных файлах. При этом остальные поля у таких записей могут отличаться.

Из-за этого я не использовал эти поля как готовые первичные ключи измерений. Вместо этого в скрипте `03_dml_transform.sql` создается временная таблица `source_keys`. В ней для одинаковых наборов полей выбирается минимальный `source_row_id`, и он используется как простой технический ключ исходной сущности.

Такой подход позволяет:
* не потерять строки продаж;
* не объединять разные объекты только из-за одинакового номера в CSV;
* сохранить связь каждой продажи с правильными измерениями.

## Скрипты
`01_init_source.sql`:
создает исходную таблицу и загружает данные из CSV.

`02_ddl_snowflake.sql`:
создает таблицы фактов, измерений и справочников.

`03_dml_transform.sql`:
заполняет справочники, измерения и таблицу фактов из `source_data`.

## Проверка результата
После выполнения скриптов были проверены основные условия:

```text
source_data = 10000 строк
fact_sales = 10000 строк
фактов без связей с измерениями = 0
дублей source_row_id в fact_sales = 0
```

То есть все исходные строки были загружены и попали в таблицу фактов, а связи с измерениями построились корректно.
